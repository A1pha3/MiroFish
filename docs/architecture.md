## 架构概览

### 设计哲学

MiroFish 的架构设计遵循以下核心哲学：

| 哲学 | 说明 | 实现方式 |
|------|------|----------|
| **关注点分离** | 不同层次处理不同复杂度的逻辑 | 分层架构 |
| **依赖倒置** | 高层模块不依赖低层模块细节 | 接口抽象 |
| **开闭原则** | 对扩展开放，对修改封闭 | 模块化设计 |
| **单一职责** | 每个组件只负责一个功能 | 服务边界清晰 |
| **接口隔离** | 通过接口而非实现进行交互 | API 设计 |

#### 核心组件

**1. OntologyGenerator (本体生成器)**

| 功能 | 说明 | 技术细节 |
|------|------|----------|
| 文本分析 | 分析文本内容和模拟需求 | 使用 LLM 的上下文理解能力 |
| 实体类型设计 | 生成 10 个实体类型 | 强制包含 Person 和 Organization 兜底类型 |
| 关系类型设计 | 生成实体间的关系类型 | 考虑社交模拟的典型关系 |

**设计决策**：为什么需要 10 个实体类型？

```
问题：不同文本内容的实体类型差异很大
├── 文学作品：人物、组织、地点、物品...
├── 舆情事件：人物、媒体、机构、事件...
└── 金融场景：公司、分析师、监管机构...

解决方案：分层设计
├── 兜底类型（2个）：Person、Organization
│   └── 捕获所有不属于具体类型的实体
└── 具体类型（8个）：根据内容定制
    └── 提高模拟的准确性和丰富度
```

**2. GraphBuilder (图谱构建器)**

| 功能 | 说明 | 技术细节 |
|------|------|----------|
| 异步构建 | 不阻塞 API 响应 | 后台线程 + TaskManager |
| 分块处理 | 处理大文本 | 默认 500 字符/块，50 字符重叠 |
| 批量 API 调用 | 提高效率 | 默认 3 块/批 |
| 进度跟踪 | 任务状态管理 |轮询或 SSE 推送 |

**数据流时序图**：

```
用户请求 ──▶ API(generate_ontology) ──▶ 返回 task_id
                                                │
                                                ▼
                            后台线程开始工作
                                                │
              ┌────────────────────────────────┐
              │ 1. 创建图谱会话               │
              │     (Zep Cloud API)            │
              └────────────────────────────────┘
                                                │
                                                ▼
              ┌────────────────────────────────┐
              │ 2. 分块处理文本                 │
              │    chunks = split_text(text)      │
              └────────────────────────────────┘
                                                │
                                                ▼
              ┌────────────────────────────────┐
              │ 3. 批量添加 episodes           │
              │    for batch in chunks:          │
              │        session.add_episodes()   │
              └────────────────────────────────┘
                                                │
                                                ▼
              ┌────────────────────────────────┐
              │ 4. 更新任务状态                 │
              │    task_manager.complete(task_id) │
              └────────────────────────────────┘
```

**3. ZepEntityReader (实体读取器)**

| 功能 | 说明 | 技术细节 |
|------|------|----------|
| 实体过滤 | 只返回符合条件的实体 | 支持按类型过滤 |
| 关系丰富 | 包含实体的关系信息 | enrich_with_edges 参数 |
| 类型统计 | 统计实体类型分布 | 用于前端展示 |

### 添加新的 Report Agent 工具

```python
# 1. 定义工具函数
from typing import Dict, Any
from app.services.zep_tools import ZepToolsService

class CustomAnalyticsTool:
    def __init__(self, graph_id: str):
        self.graph_id = graph_id
        self.zep_service = ZepToolsService(graph_id)

    def trend_analysis(self, query: str) -> Dict[str, Any]:
        """趋势分析工具"""
        # 1. 获取时间序列数据
        time_series = self._get_time_series_data(query)

        # 2. 计算趋势指标
        trends = self._calculate_trends(time_series)

        # 3. 生成洞察
        insights = self._generate_insights(trends)

        return {
            "query": query,
            "trends": trends,
            "insights": insights
        }

# 2. 注册到 Report Agent
class ReportAgent:
    def __init__(self, ...):
        # ... 现有初始化

        # 注册自定义工具
        from .custom_tools import CustomAnalyticsTool
        self.custom_tools = CustomAnalyticsTool(self.graph_id)

    def _get_tools(self) -> Dict[str, Callable]:
        return {
            # 现有工具
            "insight_forge": self.zep_tools.insight_forge,
            "panorama_search": self.zep_tools.panorama_search,
            "quick_search": self.zep_tools.quick_search,
            "entity_lookup": self.zep_tools.entity_lookup,
            "interview_agent": self._interview_agent,

            # 新工具
            "trend_analysis": self.custom_tools.trend_analysis
        }
```
