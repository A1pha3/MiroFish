## 核心概念：为什么需要知识图谱？

### 专家思维模型

专家看到的不只是一个"存储结构"，而是：

```
┌─────────────────────────────────────────────────────────────┐
│                     碎片化文本                               │
│  "贾宝玉是贾府公子...林黛玉是贾府表小姐...贾府逐渐衰败..."  │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ❌ 人类难以直接处理
                    - 信息分散
                    - 关系隐含
                    - 无法推理
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                    知识图谱转化                              │
│  ┌─────────┐                    ┌─────────┐                  │
│  │贾宝玉   │─────喜欢──────────→│林黛玉   │                  │
│  │Person   │                    │Person   │                  │
│  └─────────┘                    └─────────┘                  │
│       │                             │                        │
│       │属于                         │属于                     │
│       ▼                             ▼                        │
│  ┌─────────┐                    ┌─────────┐                  │
│  │贾府     │←───逐渐衰败────────│家族命运 │                  │
│  │Organization│                 │概念     │                  │
│  └─────────┘                    └─────────┘                  │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ✅ 智能体可以理解推理
                    - 关系显式化
                    - 支持路径查询
                    - 可计算相似度
```

### 知识图谱的四大价值

| 价值 | 说明 | 对预测的影响 |
|------|------|-------------|
| **结构化** | 将隐式关系转化为显式图结构 | 智能体能理解"谁认识谁" |
| **可检索** | 支持复杂的图查询和路径分析 | Report Agent 能找到相关证据 |
| **动态更新** | 模拟过程中实时更新记忆 | 智能体的关系会变化 |
| **时序记忆** | 保留事件的时间线 | 可以分析因果关系 |

## GraphRAG 工作原理

### 技术流程

```
原始文本 (红楼梦第1回)
    │
    ▼
┌─────────────────────────────────────────┐
│  第一步：文本分块 (Chunking)             │
├─────────────────────────────────────────┤
│  "甄士隐梦幻识通灵..."                  │
│  "贾雨村风尘怀闺秀..."                  │
│  ↓                                       │
│  块1: 500字符                           │
│  块2: 500字符 (重叠50字符)              │
│  块3: 500字符 (重叠50字符)              │
│  ...                                    │
└─────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────┐
│  第二步：本体生成 (Ontology Generation)  │
├─────────────────────────────────────────┤
│  LLM 分析文本，识别：                    │
│                                          │
│  实体类型:                               │
│  - Person (人物)                         │
│  - Organization (组织/家族)              │
│  - Location (地点)                       │
│  - Event (事件)                          │
│                                          │
│  关系类型:                               │
│  - 家属关系 (亲子、配偶、兄弟)           │
│  - 社会关系 (朋友、主仆)                 │
│  - 情感关系 (喜欢、讨厌)                 │
└─────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────┐
│  第三步：实体提取 (Entity Extraction)    │
├─────────────────────────────────────────┤
│  对每个文本块，LLM 提取：                │
│                                          │
│  实体:                                   │
│  - name: "贾宝玉"                        │
│  - type: "Person"                        │
│  - description: "贾政之子，衔玉而生"     │
│                                          │
│  关系:                                   │
│  - source: "贾宝玉"                      │
│  - target: "贾政"                        │
│  - relation: "父子"                      │
│  - confidence: 0.95                      │
└─────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────┐
│  第四步：图谱构建 (Graph Construction)   │
├─────────────────────────────────────────┤
│  Zep Cloud 构建：                        │
│                                          │
│  Nodes (节点):                           │
│  - 每个实体一个节点                      │
│  - 存储属性和描述                        │
│                                          │
│  Edges (边):                             │
│  - 每个关系一条边                        │
│  - 支持权重和方向                        │
│                                          │
│  Communities (社群):                     │
│  - 自动检测紧密连接的节点群              │
│  - 用于理解群体结构                      │
└─────────────────────────────────────────┘
    │
    ▼
  知识图谱 (可检索、可更新、可推理)
```

### 关键参数影响

```
┌─────────────────────────────────────────────────────────────┐
│                    分块参数权衡                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  小块 (300-400字符)                    大块 (800-1000字符)   │
│  ┌─────────────────┐                ┌─────────────────┐    │
│  │ 优点:           │                │ 优点:           │    │
│  │ · 提取更精细    │                │ · 上下文完整    │    │
│  │ · 跨实体关系少  │                │ · 跨块关系少    │    │
│  │                 │                │ · 调用次数少    │    │
│  │ 缺点:           │                │                 │    │
│  │ · 上下文断裂    │                │ 缺点:           │    │
│  │ · API 调用多    │                │ · 可能漏提取    │    │
│  │ · 关系可能丢失  │                │ · 细节丢失      │    │
│  └─────────────────┘                └─────────────────┘    │
│                                                              │
│  推荐策略:                                                   │
│  - 文本复杂、人物多 → 使用小块 (400-500)                     │
│  - 文本简单、叙事线单一 → 使用大块 (800-1000)                │
│  - 不确定时使用默认值 (500)                                  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 1.4 生成本体

#### 什么是本体？

本体（Ontology）定义了知识图谱的**类型系统**，类似于数据库的 Schema：

```json
{
  "entity_types": [
    {
      "name": "Person",
      "description": "具有独立人格的人物",
      "examples": ["贾宝玉", "林黛玉", "薛宝钗"]
    },
    {
      "name": "Organization",
      "description": "组织、家族、机构",
      "examples": ["贾府", "荣国府", "朝廷"]
    },
    {
      "name": "Location",
      "description": "地理位置、场所",
      "examples": ["大观园", "荣国府", "北京"]
    }
  ],
  "relationship_types": [
    {
      "name": "亲子关系",
      "description": "父母与子女之间的关系",
      "direction": "bidirectional"
    },
    {
      "name": "情感关系",
      "description": "喜欢、爱慕、讨厌等情感",
      "direction": "directed"
    }
  ]
}
```

│  □ 分析文本样本...                      │

#### 本体质量检查

生成的本体后，检查以下要点：

| 检查项 | 说明 | 例子 |
|--------|------|------|
| **类型完整性** | 是否覆盖主要实体类型 | 有 Person、Organization、Location |
| **类型独立性** | 类型之间是否重叠较少 | Person 和 Organization 应该区分 |
| **关系清晰性** | 关系描述是否明确 | "喜欢"比"有关系"更明确 |
| **示例恰当** | 示例是否典型 | 贾宝玉作为 Person 的例子很好 |

### 分块参数调优

#### 参数对照表

| 参数 | 默认值 | 调小效果 | 调大效果 | 推荐场景 |
|------|--------|----------|----------|----------|
| **块大小** | 500 | 提取更精细，API 调用多 | 上下文完整，可能漏细节 | 复杂文本→400，简单文本→800 |
| **重叠大小** | 50 | 边界信息可能丢失 | 重复信息多 | 块小时用小重叠(20-30) |
| **最大实体数** | 500 | 只保留主要实体 | 可能包含次要实体 | 关注重点人物→200，全景分析→1000 |

 │检查文件编码│       │关系太少？   │       │并行度不够？ │
 │重新构建    │       │  → 材料简单 │       │  → 增加并行 │
 └─────────────┘       │  → 补充材料 │       │LLM 太慢？   │
                       └─────────────┘       │  → 切换模型 │
                                             └─────────────┘
```

### 常见问题速解

| 问题 | 可能原因 | 快速解决 |
|------|----------|----------|
| **构建失败** | API Key 无效 | 检查 .env 配置 |
| **节点数量少** | 文本过于简单 | 补充相关材料 |
| **关系稀疏** | 分块太大 | 减小块大小到 300-400 |
| **构建超时** | 文件过大 | 分段处理，限制最大实体数 |
| **类型混乱** | 自动生成问题 | 手动编辑本体 |

```
┌─────────────────────────────────────────────────────────────┐
│            图谱质量对最终预测效果的影响                      │

│  · 太密集       → 关系混乱                → 难以提取模式    │
│                                                              │
│  类型多样性:                                                 │
│  · 单一类型     → 互动模式单一          → 预测维度有限    │
│  · 多种类型     → 互动模式多样            → 预测更全面      │
│                                                              │
└─────────────────────────────────────────────────────────────┘
